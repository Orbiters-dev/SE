**잘못한 점:**
- 스크립트 출력 "✓ 처리할 주문이 없습니다"만 보고 그대로 보고
- 03_order_list.png 스크린샷을 안 봄 → 실제로는 R-Login 화면에 막혀있었음
- mainmenu URL 도달 검증 없이 "✓ 로그인 완료" 출력하는 vacuous 성공 패턴

**Why:** [백그라운드 결과 먼저 확인](feedback_verify_bg_before_report.md) [결과 검증 후 보고](feedback_verify_before_reporting.md) [Vacuous PASS 금지](feedback_no_vacuous_pass.md) 3개 메모리 다 어김.

**How to apply:** 0건/실패 결과 보고 전 반드시 스크린샷 1장 이상 직접 Read해서 화면이 실제로 그 상태인지 확인. URL이 mainmenu/order-list-sc 류가 아니면 절대 "처리 완료/0건" 단정 금지.

## 2. 보고 시 주문번호 1:1 비교 누락 — 세은이 묻고 나서야 시작

**상황:** Auditor "ALL PASS Score 100/100" 결과 보고 후, 세은이 "오늘 한 거 다 교차 검증까지 했어?? 정확하게 옵션코드랑 송장 번호 전부 입력된 거야?" 물어봄. 그제야 STEP 1 처리 15건 vs Auditor 검증 15건을 1:1 비교 → 0458424269 누락 + 0454022785 추가 발견.

**잘못한 점:**
- Auditor 표만 보고 "15건 ALL PASS"로 만족
- STEP 1 주문번호 리스트와 Auditor 검증 리스트를 처음부터 직접 비교 안 함

**Why:** [대조 작업 대충 넘기지 마](feedback_never_skip_verification.md) "전건 1:1 비교" 룰 어김.

**How to apply:** Auditor 같은 검증 도구가 PASS 줘도 그 검증 대상 리스트와 원본 처리 리스트의 ID를 직접 1:1 매칭. 건수만 같다고 안심 금지 — 누락+추가가 동시에 일어나면 건수는 같지만 다른 주문이 처리될 수 있음.

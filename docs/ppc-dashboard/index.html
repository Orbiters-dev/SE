Executor: 9-Stage Pipeline
Pre-flight
DataKeeper Freshness Check
Verify latest data >= T-2. If stale, auto-trigger collection for 3 channels.
Stage 1-2
Profile & Campaign Data
Load brand config (budget, ACOS targets) + fetch all ENABLED campaigns from Amazon Ads API.
Stage 3
Campaign Analysis (ROAS Framework)
Apply 7 ROAS rules to each campaign. Generate proposals: pause, reduce_bid, optimize, increase_budget. Detect anomalies.
ROAS < 1.0 → pause | 1.0-1.5 → bid -30% | 1.5-2.0 → bid -15%
2.0-2.5 → optimize -10% | 2.5-3.0 → optimize | 3.0-5.0 → budget +20% | >5.0 → bid +10%, budget +30%
Stage 4
Search Term & Keyword Reports
Primary: DataKeeper cache (limit=50K). Fallback: direct API with 7d chunks. Load execution history for dedup.
Stage 5
Simulator + DataForSEO + Keyword Analysis
Load backtest JSON → boost negate confidence. Cross-reference Google CPC for bid ceiling. Generate harvest/negate/bid proposals.
Harvest: ACOS < target, 10+ clicks, 1+ sale → exact match
Negate: $0 sales + high spend, ACOS > 2x target, SIM confirmed ($10+ saveable)
Bids: strong → +20%, inefficient → -20%, no sales → -30%
Stage 6
Keyword Performance Matrix
7d vs 30d trends, cannibalization detection, Google CPC benchmarking.
Stage 7-8
Cross-Platform + ASIN P&L
Google/Meta/Shopify context. Full-cost margin per ASIN (COGS + FBA + Export + Returns). Grosmimi: 7 product categories.
Stage 9
Save & Email
JSON proposal saved. HTML email with all sections sent via Gmail OAuth. Budget impact summary included.

Module 1: Wasted Spend Detection

🗑️ How Waste Detection Works

Scans every search term across 14-day rolling windows. If a term accumulates $5+ spend with zero conversions, it gets flagged. The negative is "applied" 14 days after the threshold breach — simulating a realistic reaction time.
Spend Threshold
WASTE_SPEND_THRESHOLD = $5.00
Minimum spend in a 14d window to flag as waste
Reaction Window
WASTE_WINDOW_DAYS = 14
Days to wait before simulating negative addition
Savings Calculation
spend_after_negate - 0 = saved
All spend after negate date would have been prevented

Module 2: Bid Efficiency Optimization

⚖️ How Bid Rules Work

Retroactively applies ROAS-based bid adjustments to each keyword. Under-performers get bid reduction (lower CPC = lower cost, same conv). Scalable keywords get bid increase (more impressions = more revenue).
Under-performer Rule
ROAS < 2.0x → Bid -20%
New CPC = current × 0.80. Same conversions, 20% less cost.
Scalable Rule
ROAS > 5.0x → Bid +15%
More impressions = proportionally more revenue at high margin.
Minimum Data
$1+ spend, 2+ clicks
Skip keywords with insufficient data for reliable signals.

How Simulator Feeds into Executor

🔗 Integration Flow

1. Load Backtest JSON
_load_simulator_backtest(brand_key) — finds most recent file, max 14d old. Extracts waste terms + bid data.
2. Boost Negate Confidence
Simulator's top_waste_terms with would_save >= $10 → added as SIM confirmed negates, even if executor threshold not met.
3. Annotate Existing Negates
Existing negate proposals get sim_would_save and sim_windows appended. Sort order uses sim savings.
4. Email Rendering
SIM badge shown next to confirmed terms. Backtest Insights section with waste savings + ROAS improvement. Source file + age logged.

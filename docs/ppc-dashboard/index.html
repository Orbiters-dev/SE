Backtest Simulator — Deep Dive

Retroactive rule application engine. Answers: "What if 퍼포마 had been running from Day 1?"

🧪 Overview

The simulator pulls historical search term + keyword data from DataKeeper (PG), applies 퍼포마's optimization rules retroactively, and calculates how much spend could have been saved and how ROAS would have improved. It runs weekly (Monday KST 23:00) via ppc_simulator.yml for all 3 brands with a 30-day lookback.

🗑️ Module 1: Wasted Spend Backtest

Question: "How much money was spent on search terms that never converted, and how much could we have saved by adding them as negative keywords sooner?"
Step 1
Group by (campaign, searchTerm)
All search term rows are grouped by their unique (campaignId, searchTerm) pair. Each group contains multiple 7-day windows sorted chronologically.
Step 2
Track cumulative zero-conversion spend
For each term, scan windows in order. If purchases14d == 0, add the window's cost to a running counter. If a conversion occurs, reset counter to $0.
Step 3
Trigger negative at threshold
When cumulative zero-conv spend reaches $5.00 (WASTE_SPEND_THRESHOLD), mark the term as "negated" at that window's date. The spend in THIS window already happened (14-day rule), so savings start from the NEXT window.
Step 4
Calculate savings
All subsequent windows after the negation date → their spend is counted as "would_save". This is money that would NOT have been spent if the negative keyword had been added.
Threshold
$5.00
Cumulative spend with 0 conversions before triggering negative
Window
14 days
Wait period before negative takes effect (conservative)
Reset
Any conversion
If the term converts even once, counter resets — it's not wasteful
Save timing
Next window
Current window spend already committed; savings start from next chunk
Example: "baby straw cup" spent $1,205 over 6 windows. After $5+ with 0 conversions hit on Mar 7, it would be negated. The remaining $948 in later windows = savings.
Key insight: Terms with high total spend but some conversions can still be flagged — the counter resets on conversion, but if a long zero-conv streak follows, the threshold triggers again.

⚖️ Module 2: Bid Efficiency Backtest

Question: "If we had adjusted bids based on ROAS performance, how much better would our overall ROAS be?"
Step 1
Aggregate keywords across windows
All keyword rows are grouped by keyword ID. Spend, sales, clicks, impressions, purchases summed across all windows. Keywords with <$1 spend or <2 clicks are excluded (insufficient data).
Step 2
Calculate per-keyword ROAS
ROAS = total_sales / total_spend. Also compute CPC = spend/clicks for bid simulation.
Step 3A — Under-performers
ROAS < 2.0x → Reduce bid 20%
Logic: New CPC = CPC × 0.80. Clicks stay same (conservative). New spend = new_CPC × clicks. Sales unchanged (same conversion rate). Net effect: spend drops 20%, ROAS improves mechanically.
Step 3B — Scalable winners
ROAS > 5.0x → Increase bid 15%
Logic: Scale factor = 1.15. New spend = spend × 1.15. New sales = sales × 1.15 (linear assumption: more impressions → proportionally more clicks → more sales). Extra profit = (new_margin) - (old_margin).
Step 3C — In-range
2.0x ≤ ROAS ≤ 5.0x → No change
These keywords are performing within acceptable bounds. Spend and sales pass through unchanged to the simulated totals.
ROAS < 2.0x
Bid −20%
Spending too much relative to returns. CPC cut reduces waste.
ROAS > 5.0x
Bid +15%
High-efficiency keyword. More spend = proportionally more profit.
2.0–5.0x
Hold
Healthy range. No action needed.
Min data
$1 + 2 clicks
Keywords below this are excluded — not enough signal.
Assumption: Bid efficiency simulation assumes a linear spend-impression relationship (i.e., 20% lower bid → ~same impressions at lower CPC). In practice, lower bids may reduce impression share, so actual savings could be smaller. For scalable keywords, the +15% assumes proportional growth, which holds well for non-saturated keywords but may overestimate for already-dominant terms.

🔗 SIM → Executor Integration

Simulator results feed directly into the executor's proposal pipeline.
Feed 1
top_waste_terms → Negative proposals
Terms with would_save > $10 get tagged with SIM badge in the proposal email. Sorted by savings descending. The executor cross-references these against already-negated terms (execution history dedup).
Feed 2
Backtest insights → Email section
The proposal email includes a green "Backtest Insights" card showing potential savings, ROAS improvement, and top wasteful terms. This gives context to the campaign-level proposals.
Feed 3
Execution history → Report annotation
The backtest HTML report loads ppc_executed_*.json files and shows all previously executed changes (harvests, negatives, bid adjustments) with dates, status, and baseline metrics.
Feed 4
Communicator email
The 12-hour communicator email includes a summary table of all 3 brands' backtest results (waste savings, ROAS improvement) sourced from the latest *_backtest_*.json files.

⚠️ Assumptions & Limitations

Attribution
14-day window
Amazon's purchases14d means a click today can convert up to 14 days later. Recent windows may have incomplete attribution.
Linearity
Bid ↔ Spend
Assumes bid changes translate proportionally to spend changes. In reality, auction dynamics, competition, and quality score affect this non-linearly.
7-day chunks
SUMMARY mode
Amazon API returns aggregated data per 7-day window, not daily. Intra-week patterns are invisible — a term that converts on day 1 but wastes on days 2-7 appears as "has conversions".
No cross-term
effects
Negating one term may redirect traffic to other terms. This second-order effect is not modeled. In practice, some "saved" spend would shift to related queries.
Static market
assumption
Simulation assumes competitors, prices, and demand remain constant. Actual results vary with market dynamics and seasonality.
